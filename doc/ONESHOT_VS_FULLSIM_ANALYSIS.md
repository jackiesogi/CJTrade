# Oneshot vs Full\_sim 回測邏輯差異深度分析

> 生成日期：2026-04-15
> 分析資料：`poc/1.log` (oneshot, 172 筆) vs `poc/2.log` (full\_sim, 53 筆)

---

## TL;DR

兩套系統的 BB 參數完全一致（`bb_window_size=100`, K 線 interval=`1m`），
交易次數差距 3.2 倍的原因**純粹來自交易執行邏輯的不同**——而非參數設定。

更重要的是：**full\_sim 的績效遠優於 oneshot**，
這迫使我們重新檢視「oneshot 的邏輯一定比較正確」這個直覺。

---

## 1. 兩套系統的執行邏輯差異

### 1.1 信號產生層

兩者的 Bollinger Band 計算完全一致：

| 項目 | Oneshot (`bb.py`) | Full\_sim (`cjtrade_system_arenax.py`) |
|---|---|---|
| BB window | 100 | 100 |
| BB std dev | 2σ | 2σ |
| K 線 interval | 1m | 1m |
| 資料來源 | `ArenaXMiddleWare.get_kbars()` | `get_snapshots()` → `price_history` |

### 1.2 下單決策層（核心差異所在）

#### Oneshot：「一進一出、嚴格互斥」

```python
# bb.py 第 77-98 行
if price <= lower and not ctx.has_position(bar.symbol):   # ← 有持倉就不買
    qty = ctx.calc_qty(price, self._max_pct)               # ← equity * 5% / price
    return BUY signal

elif price >= upper and ctx.has_position(bar.symbol):      # ← 沒持倉就不賣
    qty = ctx.position_qty(bar.symbol)                      # ← 一次全部平倉
    return SELL signal
```

行為模式：`BUY → SELL → BUY → SELL → ...` 嚴格交替

#### Full\_sim：「允許加碼、分批出場」

```python
# cjtrade_system_arenax.py 第 863-906 行
if signal == 'BUY':                                        # ← 不檢查是否已持有
    quantity = self.calculate_position_size(price)           # ← equity * 5% / price
    self.client.buy_stock(...)

elif signal == 'SELL' and has_position:
    quantity = min(calculate_position_size(price),           # ← 只賣一部分
                   position.quantity)
    self.client.sell_stock(...)
```

行為模式：`BUY → BUY → BUY → ... → SELL → SELL → ...` 可堆疊

### 1.3 信號去重

| 項目 | Oneshot | Full\_sim |
|---|---|---|
| 去重機制 | 無（每根 K 線獨立判斷） | `new_signal != old_signal`（同方向信號不重複觸發） |
| 效果 | 因 `has_position` guard 間接去重 | 顯式去重但允許 stacking |

### 1.4 差異總結表

| 維度 | Oneshot | Full\_sim |
|---|---|---|
| 同 symbol 最大持倉 | **1 個** | **無限制**（實測最高 8 個） |
| BUY 前檢查 | `not has_position` | 不檢查 |
| SELL 數量 | 全部平倉 | `min(position_size, holding)` |
| 每筆交易金額 | ~150K (equity 5%) | ~250K (equity 5%) |
| 資金最大佔用 | 749K (25%) | 4,661K (93%) |

## 2. 績效實證

以 `poc/1.log` 和 `poc/2.log` 的實際數據：

### 2.1 已實現損益

| 指標 | Oneshot | Full\_sim | Full\_sim 優勢 |
|---|---|---|---|
| 完成交易數 | 84 | 28 | — |
| 勝率 | 73.8% | **92.9%** | +19.1% |

### 2.2 交易成本衝擊

以台股手續費 0.1425% + 證交稅 0.3% 計算：

| 成本項目 | Oneshot | Full\_sim |
|---|---|---|
| 總交易量 | 25,674K | 12,572K |

| 證交稅 | 37,690 | 12,313 |
| **總成本** | **74,276** | **30,227** |

**Oneshot 的交易成本 (74K) 已超過其已實現利潤 (47K)。**
計入交易成本後，oneshot 是淨虧損的。

### 2.3 為什麼 Oneshot 績效差？

Oneshot 的「一進一出」邏輯在 BB 均值回歸策略下有致命缺陷：

1. **過早平倉**：碰到 upper band 就全部賣掉，但趨勢可能才剛開始
2. **高頻交易吃成本**：84 次 round-trip，每次利潤只有 558 元，扣完手續費是負的
3. **平均持有 1.36 天**：BB window=100 根 1 分鐘 K 線，捕捉的是短期波動，但 1 天內的波幅扣掉交易成本後幾乎無利可圖
4. **虧損不對稱**：贏的時候賺 1,849，輸的時候虧 3,082 → 盈虧比 0.6:1

## 3. Position Stacking 不適用於量化交易嗎？

### 3.1 現實中的分批建倉/平倉

**分批買入（Scaling In）和分批賣出（Scaling Out）是非常主流的量化策略**，絕對適用：

| 策略 | 說明 | 使用場景 |
|---|---|---|
| Grid Trading（網格交易） | 每跌 N% 加買一批，每漲 N% 賣一批 | 震盪盤 |
| Dollar-Cost Averaging | 定期定額分批買入 | 長期投資 |
| Pyramiding | 趨勢確認後逐步加碼 | 趨勢跟蹤 |
| Scaling Out | 到目標價分 2-3 批出場 | 鎖定部分利潤 |
| Martingale / Anti-Martingale | 虧損/獲利時加碼 | 高風險策略 |

### 3.2 Full\_sim 的 Stacking 為什麼能贏

Full\_sim 的 stacking 在 BB 均值回歸策略下恰好模擬了一種**逢低加碼**行為：

```
價格在 lower band 附近震盪 → 每次觸碰都加買一批
→ 建立了較大的平均成本部位
→ 當價格最終回歸 middle/upper band 時
→ 整批部位都獲利

例：2890 在 22.90 附近買了 4 次（共 ~1M）
    → 漲到 23.40-23.75 時分批賣出
    → 每批都賺 2-3%，遠大於交易成本
```

這本質上就是一種 **grid trading / averaging down** 策略。

### 3.3 但無控制的 Stacking 有嚴重風險

Full\_sim 目前的 stacking 之所以「目前表現好」，是因為：

1. **回測期間沒有遇到大幅下跌**：如果 2890 從 22.90 跌到 18.00，8 筆 BUY 全部套牢，虧損 = 8 × 250K × 21% = **420K**
2. **沒有最大持倉限制**：資金 93% 都投入了，完全沒有應急緩衝
3. **選擇性偏誤**：BB 均值回歸策略天然假設「價格會回來」，在趨勢行情中這個假設會崩潰

#### 分批建倉的正確做法 vs Full\_sim 目前的做法

| 維度 | 正確的分批建倉 | Full\_sim 目前 |
|---|---|---|
| 最大加碼次數 | 有限（如 3-5 次） | **無限制** |
| 加碼間距 | 每跌 N% 才加碼 | 每次信號觸發就加碼（可能間隔幾分鐘） |
| 單一標的最大曝險 | 如 equity 20% | **無限制**（實測佔到 93%） |
| 總持倉上限 | 如 equity 80% | **無限制** |
| 停損機制 | 有（如跌破 BB -3σ 或 -10%） | **沒有** |

## 4. 那全部平倉真的適合嗎？

**不適合。** 以下是為什麼：

### 4.1 全部平倉的問題

Oneshot 碰到 upper band 就全部賣光，這意味著：

- **放棄了趨勢利潤**：如果一支股票從 lower band 開始上漲到 upper band 之後繼續漲（趨勢行情），oneshot 在第一次碰 upper 就出場了，錯過後面 80% 的漲幅
- **頻繁進出**：在震盪盤中，價格反覆穿越 BB → 反覆 BUY/SELL → 每次只賺一點點 → 交易成本吃光利潤
- **勝率高但盈虧比差**：73.8% 勝率但平均贏 1,849 / 平均虧 3,082 → 數學期望值幾乎為零

### 4.2 更合理的出場策略

| 方式 | 說明 | 適用場景 |
|---|---|---|
| **分批平倉**（推薦） | 碰 upper band 先賣 50%，繼續漲再賣 25%，剩 25% 留著跑趨勢 | 兼顧鎖利 + 趨勢 |
| **Trailing Stop** | 不在固定價位出場，而是設動態停利（如從高點回撤 2% 就出場） | 趨勢跟蹤 |
| **Time-based Exit** | 持有 N 天後強制出場 | 控制持倉時間 |
| **Signal Reversal** | 等到 BB 信號從 BUY → SELL 再出場（目前 full\_sim 的做法） | 均值回歸 |

## 5. 結論與建議

### 5.1 兩套系統各有缺陷

| | Oneshot 的問題 | Full\_sim 的問題 |
|---|---|---|
| 進場 | ✅ 單次進場風險可控 | ⚠️ 無限加碼無上限 |
| 出場 | ❌ 全部平倉太早退出 | ⚠️ 分批出場概念對但量不對 |
| 資金管理 | ⚠️ 只用 25% 資金，效率低 | ❌ 93% 資金鎖死，無緩衝 |
| 交易成本 | ❌ 高頻交易成本 > 利潤 | ✅ 低頻交易成本可控 |
| 風險控制 | ✅ 單一標的曝險有限 | ❌ 無最大曝險限制 |

### 5.2 理想的混合方案

結合兩者優點的策略邏輯：

```
進場：
  - 允許分批建倉（full_sim 概念）
  - 但限制：同一 symbol 最多加碼 N 次（如 3 次）
  - 且限制：單一 symbol 總曝險不超過 equity 的 M%（如 15%）
  - 且限制：總持倉不超過 equity 的 P%（如 80%）
  - 加碼間距：上一次買入後至少跌 X% 才能再加碼

出場：
  - 分批平倉（不是一次全賣）
  - 碰 upper band：賣出 50% 持倉
  - 碰 upper band + 1σ：再賣 30%
  - Trailing stop 管理剩餘 20%
  - 或：持有超過 T 天強制平倉

停損：
  - 單筆部位虧損超過 L% 就停損（如 -8%）
  - 整體持倉虧損超過 equity 的 D% 就全部停損（如 -5%）
```

### 5.3 關於架構分層

不論採用哪種策略邏輯：

- **Strategy 層**（`bb.py`）：只負責信號產生（BUY / SELL / HOLD），不管帳戶
- **Engine/System 層**：負責信號過濾（去重、持倉檢查）、position sizing、風控
- 這樣同一個 Strategy 被 oneshot 和 full\_sim 調用時，行為才會一致
